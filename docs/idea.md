# FireScribe 初版 Idea

更新时间：2026-06-29

## 一句话定位

FireScribe 是一个面向扫描手写文章的转录、审校和归档工作台，通过 OpenAI 兼容 OCR/VLM API 生成候选电子稿，再通过高效人工校对把不可靠的识别结果变成可追溯、可检索、可导出的文本资产。

## 要解决的问题

几百页手写扫描稿的数字化难点不只是批量 OCR，而是：

- 手写识别质量不稳定，容易出现形近字、漏行、乱段、模型猜补。
- 不同 OCR/VLM 模型的结果各有优劣，缺少统一管理、对比和复核入口。
- 人工校对成本高，如果不能快速定位原图和差异，效率会被拖垮。
- 电子版文本如果无法回到原始页和区域，后续引用、审校、纠错都不可靠。
- 原始文件、识别结果、人工定稿和导出文件如果混在一起，长期管理会混乱。

## 产品原则

1. 原始文件不可变。
   导入后的原始扫描件按内容哈希归档，不覆盖、不重写。

2. 识别结果可追溯。
   每次 OCR/VLM 调用都作为独立 run 保存，记录 provider、model、prompt、参数、时间和原始返回。

3. 人工修改有版本。
   系统候选稿、人工校对稿、最终定稿分层保存，不只保留“最新文本”。

4. 文本能回到页面。
   初版先按原始页码建立追溯；区域坐标、批注和低置信标记后续再增强。

5. 自动化服务于审校效率。
   目标不是假装全自动正确，而是把人工注意力集中到最可能出错的位置。

## 目标用户与场景

- 个人或家庭手稿整理：日记、文章、信件、笔记。
- 研究者或编辑：需要长期管理、检索、引用一批扫描稿。
- 档案整理者：需要保存原件、转录文本和导出版本。

初版默认是单人 Web 工具，不优先处理多人协作、权限、云同步和复杂审批。首批内容场景以中文手写文章为主，默认是正常横排文章，不优先处理复杂版式、竖排、表格和混合排版。

## 核心工作流

```text
导入文件
  -> 原件入库
  -> PDF/图片拆页
  -> 页面图登记和缩略图生成
  -> OpenAI 兼容 OCR/VLM 识别
  -> 保存各模型原始结果
  -> 生成候选稿
  -> 人工校对
  -> 全文检索
  -> 导出电子稿
```

## 主要模块

### 1. 文档库

管理文档、标签、作者、来源、导入时间、页数、处理状态。项目/合集层级后置。

需要支持：

- 文档列表和筛选
- 标签管理
- 状态筛选：未处理、识别中、待校对、校对中、已定稿、失败
- 全文检索
- 原件和导出文件回溯

### 2. 导入与文件存储

支持导入：

- PDF
- 单张图片
- 图片目录

存储策略：

- 原始文件单独存储，不放进 SQLite。
- 使用 sha256 作为资产身份，避免重复导入。
- SQLite 存 metadata、路径、hash、页码、处理状态和索引。

### 3. 页面处理

将文档拆成标准页面图像，后续 OCR/VLM 都基于页面执行。

初版可以先做：

- 已确认首批 PDF 每页都是单独图片，按页码直接提取原始页面图（保留原始格式，不重新栅格化）
- 图片统一登记为页面
- 页面宽高记录
- 缩略图生成；页面旋转先作为查看器状态，不改写原图

后续再做：

- 自动裁边
- 去阴影
- 倾斜矫正
- 对比度增强
- 区域切分

### 4. OpenAI 兼容 OCR/VLM 识别

每个识别器实现统一接口。初版至少支持：

- 通过 OpenAI 兼容 API 接入视觉模型或 OCR/VLM 服务
- 配置 base URL、API key、model、prompt version 和模型参数

识别结果分层：

- raw result：每个 provider 原始返回
- candidate text：系统合并或用户选择后的候选稿
- final text：人工确认后的最终文本

初版不要求 OCR 模型输出复杂位置结构，避免增加识别负担。页面追溯先按原始页码建立，provider 返回的坐标、置信度、版面信息等原始元数据仍应保留在 raw result 中，供后续增强使用。

### 5. 校对工作台

这是 FireScribe 的核心界面。

建议布局：

- 左侧：扫描页查看器
- 右侧：文本编辑器
- 顶部：模型结果切换、状态、保存、导出入口
- 底部或侧栏：识别详情、任务状态

初版需要支持：

- 按页校对
- 保存人工文本
- 快捷键切换上一页/下一页
- 查看不同 OCR/VLM 的原始文本
- 每页全链路状态面板（识别次数、最近模型、最高置信度、是否有定稿）

后续增强：

- 文本和原图区域联动
- 行/段级差异对比
- 低置信队列
- 页级批注和存疑标记
- 区域批注
- 审校历史 diff

### 6. 批注系统（后续）

MVP 暂不实现批注系统和存疑标记，后续再做：

批注类型：

- 页级批注
- 文本范围批注
- 区域批注
- 存疑字词
- 处理备注

批注应有状态：

- open
- resolved
- ignored

### 7. 导出

初版导出：

- TXT
- Markdown

后续导出：

- DOCX
- PDF 审校版
- 带批注版本
- 干净定稿版本

导出选项：

- 是否保留页码
- 是否保留存疑标记（后续）
- 是否包含批注（后续）
- 是否按文档或项目合并导出（项目层级后续）
- 输出格式可多种可选，初版先提供 TXT 和 Markdown，后续扩展 DOCX、PDF 审校版和带批注版本

## MVP 范围

第一版目标是跑通可靠闭环，而不是一次做完所有智能能力。

MVP 包含：

- 文档库基础列表
- PDF/图片导入
- 原始文件哈希存储
- 页面拆分和页面表
- 1 个 OpenAI 兼容 OCR/VLM 适配器
- 至少 1 个模型配置
- 每页识别结果保存
- 页面级候选稿
- 左图右文校对界面
- 人工定稿保存
- SQLite FTS5 全文检索
- Markdown/TXT 导出

MVP 暂不包含：

- 自训练手写模型
- 多人协作
- 云同步
- 复杂权限
- 完整版式还原
- 自动段落级对齐
- 高级图像增强
- 完全离线 OCR/VLM
- 批注系统和存疑标记
- 项目/合集层级

## 成功指标

建议用一批真实扫描稿做基准集，例如 20-50 页，其中人工精校 3-5 页作为评测标准。

核心指标：

- 每页从导入到候选稿的处理耗时
- 每页人工校对耗时
- 低置信项命中率
- 字符错误率或人工修改量
- 漏行、乱序、模型猜补次数
- 用户每小时能确认的页数

初版达标目标可以设为：

- 对清晰扫描稿生成可用候选稿。
- 人工校对效率明显高于从零录入。
- 每个最终文本都能追溯到原始页面和识别来源。

## 后续方向

- 多模型差异视图：把不同 OCR/VLM 的分歧直接暴露给审校者。
- 低置信队列：优先处理最可能出错的字词、行、段。
- 作者笔迹档案：按作者保存识别效果、常见误识别、专有词表。
- Prompt 版本管理：对 VLM 提示词进行版本化和 A/B 测试。
- 训练数据积累：把人工校对结果沉淀为未来微调或评测数据。
- 插件化识别器：允许用户添加新的 OCR/VLM provider。
- 项目/合集层级：支持跨文档组织和合并导出。
